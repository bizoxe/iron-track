## Выбор RDBMS

#### КОНТЕКСТ
Критерии выбора RDBMS (на основе Database Schema Blueprint):
- Обеспечение строгой реляционной целостности;
- Поддержка кастомных типов (ENUM) и часовых поясов (Timezones);
- Поддержка UUID v7 или механизмов хранения сортируемых UUID;
- Возможность создания сложных уникальных индексов и индексных выражений (Functional Indexes);
- Наличие продвинутых ограничений (Check Constraints);
- Аналитические возможности: Generated Columns, оконные функции, потенциальное использование триггерной логики;
- Возможность эффективного хранения и поиска данных в формате JSON (как расширяемая альтернатива для неструктурированных данных);
- Экосистема и производительность: нативная поддержка высокопроизводительных асинхронных драйверов (asyncpg) и зрелый инструментарий для профилирования сложных запросов.

#### РЕШЕНИЕ
Выбранная RDBMS: PostgreSQL

#### КОМПРОМИССЫ И ОГРАНИЧЕНИЯ
- Connection Overhead: в отличие от MySQL (Thread-per-connection), Postgres создает отдельный процесс ОС для каждого соединения (Process-per-connection) — это дорого с точки зрения памяти;
- VACUUM & MVCC: требует мониторинга процессов `autovacuum` для предотвращения раздувания таблиц (bloat) при высоких нагрузках на запись/удаление;
- Сложность настройки: по умолчанию заданы очень консервативные настройки (shared_buffers, work_mem и т.д.), которые необходимо оптимизировать вручную.

#### ОБОСНОВАНИЕ И РАССМОТРЕННЫЕ АЛЬТЕРНАТИВЫ
Рассмотренные варианты (Лицензирование):
- MySQL (Open-source с коммерческими элементами);
- MariaDB (Open-source с коммерческими элементами);
- PostgreSQL (True Open Source);
- SQLite (True Open Source).

Отклоненные варианты:
- SQLite;
- MySQL;
- MariaDB.

Основные причины:
Ни одна из альтернативных СУБД не обеспечивает необходимой гибкости и полноты требований, заложенных в схему (в частности: сложные функциональные индексы, оконные функции и строгая типизация данных), оставаясь при этом полностью Open Source решением.

#### Теория

<details>

> **Self-Note:** *SQLite*
> - Сложность горизонтального масштабирования;
> - Отсутствие встроенного контроля доступа (RBAC на уровне БД);
> - Ограничения при конкурентной записи (блокировка всей базы при write-операциях).
>
> **Note (Testing):** SQLite допустим для использования в тестах, однако это сопряжено с рисками расхождения в консистентности ограничений (Database Constraints) между тестовой и продакшн средами.

> **Self-Note:** *PostgreSQL*
> - Полное соответствие ACID;
> - ENUM в PostgreSQL — отдельный тип (хранится как 4-байтовое число); *легко добавить, но сложно удалить/переименовать*;
> - В PostgreSQL v18 появилась поддержка UUID v7; *advanced-alchemy позволяет использовать UUID v7*;
> - Поддержка частичных индексов;
> - Хорошая поддержка асинхронности; *asyncpg — один из самых быстрых в Python-стеке драйверов*;
> - Мощные Window Functions;
> - Нативная индексация выражений; *в PostgreSQL Functional Indexes и Expression Indexes используются как синонимы*
> - Позволяет хранить данные в формате JSONB;
> - CHECK — полноценный валидатор данных на уровне ядра.
>
> Также следует отметить, что PostgreSQL - своего рода "Enterprise для бедных", полноценная расширяемую платформа для управления данными с мощным планировщиком.

> **Self-Note:** *MySQL & MariaDB*
> - Ограниченная индексация: Менее гибкая работа с функциональными индексами; отсутствие нативной поддержки GIN-индексов, что делает поиск по вложенным JSON-структурам неэффективным.
> - Слабая типизация (Looser Typing): Специфичная реализация `ENUM` и работы с `Timezones` требует дополнительных усилий на уровне приложения для поддержания консистентности данных.
> - Аналитический потолок: Исторически более слабая оптимизация сложных аналитических запросов (`Window Functions`, `Recursive CTE`) по сравнению с движком PostgreSQL.
> - Лицензионные риски: Вопросы владения и лицензирования (Oracle/MariaDB Corp) делают PostgreSQL более безопасным "True Open Source" выбором для долгосрочных проектов.

</details>

## Выбор Connection Pooler

#### КОНТЕКСТ
- Пулинг на стороне клиента (внутри SQLAlchemy/asyncpg) ограничивает масштабирование:
    - Геометрическая прогрессия соединений:
        - Суть: каждый воркер приложения создает свой независимый пул;
        - Последствия: если у каждого воркера свой пул (например, `min_size=10`), то запуск 10 воркеров сразу и перманентно резервирует 100 соединений в Postgres, даже если приложение простаивает.
    - Негибкое распределение:
        - Суть: изоляция пулов на уровне процессов ОС (воркеров);
        - Последствие: внутренний пул воркера №1 не может «отдать» свои свободные соединения воркеру №2, который сейчас под нагрузкой. Соединения «заперты» внутри процессов приложения.

#### РЕШЕНИЕ
Выбранный Connection Pooler: PgBouncer  
Конфигурация:
- Pool Mode: transaction  

#### КОМПРОМИССЫ И ОГРАНИЧЕНИЯ
- Ограничения Transaction Pooling:
    - Не допускает использования сессионных возможностей, таких как временные таблицы или команды `SET`, требуя адаптации (например, использования `SET LOCAL`).
    - Непредсказуемость блокировок сессионного уровня;
    - Требуется дополнительная настройка драйвера (`statement_cache_size=0`), чтобы избежать ошибок «statement not found».
- Однопоточность (может быть проблемой для массивных Enterprise-систем);
- Отладка: может «проглатывать» или изменять специфические ошибки Postgres, выдавая общую ошибку соединения;
- Дополнительный узел в инфраструктуре, требующий отдельного мониторинга и настройки лимитов.

#### ОБОСНОВАНИЕ И РАССМОТРЕННЫЕ АЛЬТЕРНАТИВЫ
Рассмотренные варианты:
- Многопоточный Odyssey (Yandex);
- Pgpool-II «Комбайн» (Connection Pooling, Load Balancing);
- Легковесный однопоточный PgBouncer.

Отклоненные варианты:
- Odyssey;
- Pgpool-II.

Основные причины:
- Для большинства задач, включая мой проект, связки PgBouncer + PostgreSQL более чем достаточно;
- Odyssey и Pgpool-II сложнее в настройке, особенно Pgpool-II;
- PgBouncer — самый легковесный (в сравнении с Odyssey/Pgpool-II), что актуально для моего железа (AMD FX-8320 / 12 GB RAM).

#### Теория

<details>

> **Self-Note:** *Зачем нужен пулер?*  
> В отличие от MySQL, например, которая плодит легковесные потоки (threads), Postgres на каждое новое входящее соединение делает fork основного процесса (`postmaster`).
> Поэтому мы ограничиваем max_connections (default=100) и если придет 101 клиент, то получит "FATAL: sorry, too many clients already".
> Это дает изолированность (один процесс упадет - он не потащит всю базу), но как мы знаем создать/убить процесс - это дорого.
>
> Можно решить проблему "тяжелого fork" на стороне сервера БД на уровне приложения, как пример через asyncpg (asyncpg.create_pool()) или sqlalchemy (pool_size и брать соединения из QueuePool), но важно также учитывать,
> что это работает только в режиме Session Pooling. Таким образом мы будем удерживать активные соединения. Но мы сталкиваемся с другой проблемой - масштабирование и увеличение количества соединений.
> **О чем я:** если запускаем uvicorn, например, а как правило мы используем n воркеров, то каждый воркер - это отдельный процесс, у него собственный экземпляр sqlalchemy и пула соединений, то есть pool_size=20 * n - активных
> соединений к постгрес + оверхэд питон кода (управление очередью, проверка pre-ping, очистка старых коннектов).
>
> Вывод:  
> Использование пулера внутри приложения создает жесткую зависимость ресурсов БД от количества воркеров бэкенда. На инфраструктурном уровне это лишает нас гибкости при масштабировании. Внедрение внешнего пулера позволяет разорвать эту связь, превращая соединения в общий, эффективно управляемый ресурс.

> **Self-Note:** *3 режима пулера* 
> 
> - session pooling - соединение закрепляется за клиентом, пока тот не отключится (session.close());
> - transaction pooling - соединение отдается клиенту только на время транзакции;
> - statement pooling - соединение отдается на каждый отдельный SQL-запрос.
> - **Note:**   
>   - session pooling - самый безопасный, но не эффективный для масштабирования режим, не дает "уплотнения" транзакций;     
>   - transaction pooling - позволяет эффективно мультиплексировать тысячи запросов через малый пул процессов БД;    
>   - statement pooling - транзакции (BEGIN...COMMIT) в нем ломаются, так как разные строки одной транзакции могут улететь в разные процессы Postgres.    

> **Self-Note:** *Transaction Pool Mode*
> 
> С точки зрения пропускной способности всей системы `pool_mode=transaction` - наиболее производительный, с точки зрения одиночного запроса он может быть чуть медленнее, чем session pooling с включенным кэшем стейтментов.
> 
> **Проблема:** 
> *несовместимость Prepared Statements и Transaction Pooling*  
> 
> `asyncpg` (и sqlalchemy под ним) кэширует стейтменты на стороне клиента (в памяти Python-процесса) и по умолчанию готовит запрос (PREPARE stmt AS...) и пытается выполнить его по имени. В режиме pool_mode = transaction
> следующая команда может попасть на другой процесс Postgres, который про этот stmt ничего не знает. Поэтому и возникает некоторая сложность настройки подключения движка алхимии в режиме transaction.  
> 
> **Решение:**
> *в настройках sqlalchemy устанавливаем:*  
> - statement_cache_size=0 - отключаем кэш подготовленных выражений;   
> - poolclass=NullPool - каждое обращение к БД из приложения будет инициировать новое TCP/Unix-соединение до PgBouncer. 
>   - **Note:**   
>      - statement_cache_size=0 - небольшая потеря производительности, так как каждый запрос парсится базой заново.

> **Self-Note:** *Подытожим какие проблемы PgBouncer решает* 
> 
> - Защита от "протокольного шока":  
> FastAPI + asyncpg могут генерировать сотни конкурентных запросов в секунду. Даже если сервис один, он работает в событийном цикле.
> Без пулера каждое новое соединение — это тяжелый fork процесса в ОС на стороне Postgres.
>
> - Масштабирование:  
> Легко добавить новые инстансы (либо через Docker, либо через воркеры).
> Без пулера алхимия может держать открытые соединения, мы решаем проблему "тяжелого fork" на стороне сервера БД, но появляется другая проблема, при добавлении инстанса - растет кол-во соединений к постгрес, что при пиковых нагрузках
> приведет к переключению контекста.
> 
> - Инфраструктурная чистота:  
>   - Позволяет например проводить техобслуживание базы данных (есть команды приостановки пулера), то есть pgbouncer ставит соединение на паузу при этом соединение с бд не рвется;
>   - Позволяет жестко ограничить ресурсы, которые приложение может отъесть у сервера БД.
>
> **Note:**
> Легко добавить новые инстансы (либо через Docker) - через Docker - я подразумеваю, что Docker-контейнеры стучатся в один PgBouncer (централизованный пулер).

</details>

## Выбор ORM

#### КОНТЕКСТ
Критерии выбора ORM (на основе Database Schema Blueprint | PostgreSQL):
- Структурные критерии:
  - Полная поддержка продвинутых функций SQL (оконные функции, CTE) для аналитики тренировок;
  - Гранулярный контроль над загрузкой отношений (избегание N+1) и сложными джоинами/агрегациями;
  - Сложные ограничения PostgreSQL, сложные индексы (функциональные/частичные).
- Критерии поддерживаемости и целостности данных:
  - Поддержка паттерна Data Mapper для отделения бизнес-логики от хранения данных (снижение когнитивной нагрузки, улучшение тестируемости и модульности);
  - Возможность реализации логики копирования данных (снимков) между шаблонами и историей сессий на уровне ORM (через механизмы событий/хуков).

#### РЕШЕНИЕ
Выбранная ORM: Advanced Alchemy

#### КОМПРОМИССЫ И ОГРАНИЧЕНИЯ
- Advanced-Alchemy:
  - Требует изучения документации;
  - Проект относительно молодой; крупные обновления могут содержать breaking changes, требующие рефакторинга;
  - В некоторых случаях абстракции становятся избыточными, и приходится использовать SQLAlchemy напрямую для реализации сложной/специфичной логики;
  - Интеграция с FastAPI:
    - Встроенные механизмы фильтрации/сортировки могут требовать дополнительной ручной обработки, так как позволяют пробрасывать некорректные параметры (например, несуществующие колонки) в БД, что ведет к исключениям.

- SQLAlchemy:
  - Довольно высокий порог входа;
  - «Магия» ORM-слоя и поверхностное знание самой библиотеки приводят к проблеме N+1, неоптимальным запросам и т.д.;
  - Оверхед в виде ORM-слоя с одной стороны и удобство разработки с другой.

#### ОБОСНОВАНИЕ И РАССМОТРЕННЫЕ АЛЬТЕРНАТИВЫ

Рассмотренные варианты:
- Tortoise ORM (Active Record);
- SQLModel (Data Mapper);
- SQLAlchemy 2.0+ (Data Mapper (фундаментально), Active Record (можно имитировать через DeclarativeBase));
- Advanced Alchemy (Service/Repository).

Отклоненные варианты:
- Tortoise ORM;
- SQLModel.

Основные причины:    
- Обе библиотеки хороши для быстрого прототипирования, для приложений с простой бизнес-логикой и высокими требованиями к масштабируемости, где не нужен глубокий контроль над JOIN/агрегациями и нет высоких требований к аналитике (Window Functions, CTE).

**Примечание:** Совокупность требований к аналитической гибкости и строгой изоляции данных фактически делает SQLAlchemy единственным зрелым выбором в экосистеме Python, способным удовлетворить всем критериям отбора без деградации архитектуры до уровня Raw SQL.

Выбор Advanced Alchemy обусловлен:
- Масштабируемостью: для поддержания кодовой базы на приемлемом уровне (decoupling, когнитивная нагрузка) необходимо либо реализовывать паттерн Repository/DAO самостоятельно, либо использовать готовые решения;
- Автоматизацией рутины: избавляет от написания шаблонного кода для CRUD-операций;
- Сложностью реализации: слой Service/Repository в проектах со сложной бизнес-логикой требует глубоких знаний внутреннего устройства SQLAlchemy;
- Снижением рисков: использование advanced-alchemy позволяет минимизировать риск типичных ошибок проектирования (управление жизненным циклом сессии, оптимизация N+1, консистентность Identity Map).

#### Теория

<details>

> **Self-Note:** *Tortoise ORM & SQLModel*
> - Tortoise ORM — реализация паттерна Active Record, где модель (класс) — сама таблица в базе данных, а экземпляр класса — это строка в этой таблице. 
> Это означает, что бизнес-логика и логика работы с данными (сохранение, удаление, поиск) смешаны в одном месте.
> Код лаконичный, но в сложных системах часто приводит к тому, что модели становятся "слишком жирными" и их трудно поддерживать.
>
> - SQLModel (Data Mapper) — по сути обертка над SQLAlchemy с типизацией и прикрученной Pydantic; главная проблема в жесткой связке (coupling): она заставляет одну и ту же модель быть и Pydantic-схемой, и SQLAlchemy-таблицей, нарушая принцип единственной ответственности (Single Responsibility Principle) за счет смешения задач передачи данных (DTO) и доступа к данным (DAO).
> Суть паттерна Data Mapper: модели — это просто описания данных (могут быть "чистыми" классами), а за работу с базой отвечает внешний посредник — Session (Unit of Work).
> Объект user ничего не знает о базе. Чтобы его сохранить, нужно передать его в сессию: `session.add(user)`.  

> **Self-Note:** *Особенности SQLAlchemy* 
> - Золотой стандарт де-факто в крупных проектах;  
> - Мощная, зрелая и гибкая библиотека с подробной качественной документацией;  
> - Разделение на:  
>   - SQLAlchemy Core: мощный конструктор SQL-запросов. Работа с таблицами и колонками напрямую. Это "низкий уровень", максимально близкий к сырому SQL;
>   - SQLAlchemy ORM: надстройка, которая реализует Data Mapper;
> - При знании синтаксиса SQL запросы строятся как конструктор;
> - Позволяет строить сложные joins и применять window functions.

</details>

## Выбор Web-слоя

#### КОНТЕКСТ
- Ключевые возможности:
  - Интеграция с SQLAlchemy;
  - Асинхронность.
- Developer Experience:
  - Автоматическая генерация документации;
  - Типизация.
- Эффективность и обслуживание:
  - Производительность и потребление ресурсов самим фреймворком;
  - Поддержка валидации и сериализации (или относительно «дешевое» внедрение);
  - Распространенность (сообщество и найм).

#### РЕШЕНИЕ
Выбранный Web-слой: FastAPI

#### КОМПРОМИССЫ И ОГРАНИЧЕНИЯ
Ниже приведены фундаментальные особенности и ограничения фреймворка:
- Жесткая связь с Pydantic: 
  - Ввод: FastAPI использует Pydantic в самой своей основе для парсинга тела запроса, параметров пути и запроса;
  - Вывод: можно использовать, например, кастомный Response (`msgspec.json.encode`), но при этом теряется автоматическая генерация OpenAPI-схемы и оверхед от `jsonable_encoder`.
- Минимализм: 
  - Из коробки: нет ORM, нет встроенной системы миграций, нет админки, нет управления пользователями;
  - Экосистема: существует большое количество библиотек, но они поддерживаются энтузиастами.
- Оверхед и производительность:
  - Он «быстрый» в плане асинхронности (IO), но «тяжелый» в плане CPU (из-за большого количества магии под капотом при валидации).
- Внедрение зависимостей (Dependency Injection):
  - Связанность с фреймворком;
  - DI через параметры функций, а не через конструктор;
  - Отсутствие полноценного IoC-контейнера.
- Фоновые задачи (Background Tasks):
  - Подходят для простых задач (для более сложных нужно что-то вроде Taskiq).
- Сильная привязка к генерации Swagger.

#### ОБОСНОВАНИЕ И РАССМОТРЕННЫЕ АЛЬТЕРНАТИВЫ
Рассмотренные варианты:
- FastAPI;
- Litestar.

Отклоненные варианты:
- Litestar.

Основные причины:  
- Несмотря на отличную производительность и систему плагинов, Litestar остается менее распространенным решением в индустрии (на момент апреля 2026).

Выбор FastAPI обусловлен следующими факторами:
- Соответствие ключевым критериям: полностью покрывает требования по асинхронности, типизации и интеграции с выбранным стеком (SQLAlchemy).
- Актуальность (апрель 2026): фреймворк остается отраслевым стандартом и самым востребованным решением в Python-стеке, что гарантирует долгосрочную поддержку и наличие обширной базы готовых решений.
- Отказ от «коробочной магии»: отсутствие встроенных механизмов аутентификации и кэширования — осознанный выбор. Несмотря на рост трудозатрат (development cost), кастомная реализация позволяет:
  - Глубоко проработать логику безопасности (Auth/Authz) и механизмы инвалидации кэша;
  - Избежать лишнего оверхеда универсальных решений;
  - Выстроить архитектуру «от первого принципа», понимая работу каждого узла системы.

#### Теория

<details>

> **Self-Note:** *Особенности DI в FastAPI*
> 
> Синхронная природа DI FastAPI: для понимания следует разделить понятия **способ объявления** зависимости и **механизм её вызова**.
> - Пример первый: когда используем **Depends(your_func)**, FastAPI под капотом делает следующее:
>   - Если your_func определена как def: FastAPI вызывает её напрямую в отдельном потоке (worker thread) из внешнего пула (ThreadPool), чтобы не блокировать основной Event Loop;
>   - Если your_func определена как async def: FastAPI дожидается её выполнения через await.
> 
> - Пример второй: допустим есть ендпоинт -> **async def index(user=Depends(get_user), db=Depends(get_db))**, тогда:
>   - FastAPI сначала полностью разрешит get_user, и только когда она вернет результат, начнет разрешать get_db. Он не запускает их конкурентно через asyncio.gather(), даже если они обе асинхронные;
>   - В этом и заключается синхронная природа, проблема кроется в графе зависимостей, FastAPI вычисляет зависимости последовательно (sequential), а не параллельно.
> 
> - Отсутствие динамического/отложенного разрешения: 
>   - В классических DI-контейнерах (как в Java или некоторых библиотеках Python) мы можем получить "фабрику" или "прокси", который создаст объект только в момент обращения:
>     - В FastAPI Depends — это исполнитель: пришел запрос -> FastAPI смотрит на сигнатуру функции -> синхронно (последовательно) проходит по дереву зависимостей снизу вверх -> создает все объекты -> передает готовые значения в эндпоинт;
>     - Из этого следует, что время ответа эндпоинта — это **сумма времен разрешения всех его зависимостей**.
>
> **Note (AMD FX-8320)**: 
>
> Эффект накопления задержек от последовательного разрешения зависимостей особенно заметен на системах с ограниченной производительностью CPU. Для оптимизации независимые IO-bound операции лучше группировать внутри одной зависимости и запускать параллельно через `asyncio.gather`.

> **Self-Note:** *Жизненный цикл и Context Manager (yield)*
> 
> Зависимости с yield работают как жесткий контекстный менеджер: код до yield -> эндпоинт -> код после yield.
>
> Эта цепочка неразрывна и синхронизирована с HTTP-циклом. Нельзя "отпустить" выполнение пост-логики (например, коммит в БД) в фон, не задерживая при этом завершение HTTP-ответа, если это часть цепочки Depends.

> **Self-Note:** *Что такое IoC-контейнер?*
> 
> **Понятие IoC (Inversion of Control):**
>  - В обычном коде мы сами управляем созданием объектов. Например, если сервису нужен репозиторий, пишем repo = Repository() внутри сервиса. Это жесткая связь;
>  - Inversion of Control — это когда не наш код создает зависимости, а кто-то извне "отдает" их нашему коду. Мы просто говорим: "Мне нужен репозиторий", и система нам его предоставляет.
> 
> **IoC-контейнер** — это "умный склад" или менеджер объектов нашего приложения. Его задачи:
> - **Регистрация:** Мы записываем в него правила (например: "Когда просят UserRepository, создай объект класса PostgresUserRepository"); **Суть:** в сложных системах класс, который мы запрашиваем (интерфейс), и класс, который в итоге создается (реализация), — это разные сущности. Это позволяет подменять логику (например, для тестов), не меняя код в самих эндпоинтах;
> - **Управление жизненным циклом (Scope):** Контейнер решает, нужно ли создавать новый объект при каждом запросе или отдавать один и тот же (Singleton);
> - **Автоматическое разрешение зависимостей (Auto-wiring):** Если просим сервис, которому нужен репозиторий, а репозиторию нужна сессия базы данных — контейнер сам соберет всю эту цепочку и выдаст нам готовый сервис.
> 
> **Note:** В FastAPI "нет полноценного IoC-контейнера"?
> 
> В FastAPI есть встроенная система `Depends()`, но технически это **Service Locator**, работающий на уровне функций-обработчиков (эндпоинтов), а не полноценный контейнер. Вот почему его называют "неполноценным".
> 
> Пример: -> **async def get_users(repo: UserRepository = Depends()):** -> здесь UserRepository — это одновременно и "имя" зависимости, и сам код. FastAPI просто создает инстанс этого конкретного класса.
> То есть, мы просим класс A - создается инстанс класса A, нет промежуточного "умного" слоя, который мог бы гибко подменить логику этого класса во всем приложении, не меняя код в самих эндпоинтах.
> Когда опытные разработчики говорят, что в FastAPI DI "ненастоящий" или "неполноценный", они бьют именно в точку отсутствия IoC-контейнера.

</details>

## Выбор In-Memory хранилища

#### КОНТЕКСТ
- JWT Revocation: необходим черный список для refresh-токенов;
- Auth Entity Caching: хранение сериализованных объектов пользователей без дорогих запросов к реляционной БД;
- Rate Limiting (будущие задачи);
- Resource List Caching: кэширование дорогостоящих результатов пагинации (например, списки пользователей, упражнений).

**Примечание (Rate Limiting):** Это может быть реализовано через библиотеку `cashews` (через rate_limit или failover + rate_limit), так как она уже есть в стеке. Это позволит избежать раздувания количества зависимостей специализированными лимитерами (текущие требования не подразумевают сложной иерархии лимитов).

**Примечание:** Refresh-токены обычно имеют долгий цикл жизни — хранилище должно поддерживать сброс данных на диск; можно было бы выделить одно хранилище чисто под токены, но это +1 узел в инфраструктуре и дополнительная нагрузка для моего железа (AMD FX-8320). 

**Примечание:** На текущем этапе разработки я придерживаюсь того, что все закэшированные объекты имеют установленный TTL, включая блэклист токенов. Самой оптимальной стратегией вытеснения (Eviction Strategy) будет установка параметра `maxmemory-policy: volatile-ttl` и увеличение лимитов памяти.  

#### РЕШЕНИЕ
Выбранное In-Memory хранилище: Valkey

Конфигурация:
- Eviction Strategy: volatile-ttl + Увеличенный лимит памяти

**Примечание:** Для горячих данных (например, UserAuth) в качестве сериализатора/десериализатора выбран `msgspec.msgpack` для экономии ресурсов CPU и RAM.

#### КОМПРОМИССЫ И ОГРАНИЧЕНИЯ
- Потребление RAM: хранение в оперативной памяти дороже, чем на SSD/HDD, что требует внедрения TTL и политик вытеснения данных;
- Persistence (AOF): необходимость включения режима AOF для сохранности критически важных данных (например, заблокированных токенов), что снижает общую пропускную способность системы;
- Durability и лимиты железа: полная гарантия записи на диск требует вызова fsync на каждую операцию, что нивелирует преимущество in-memory решения и значительно увеличивает задержки;
- Проблема Consistency vs Availability: репликация в Redis/Valkey по умолчанию асинхронная; если мастер-узел записал данные и тут же «упал», не успев передать их реплике — данные будут потеряны;
- Сетевой оверхед: актуально, если приложение и хранилище на разных узлах (при локальном размещении возможна оптимизация через Unix-сокеты);
- Оверхед CPU при сериализации/десериализации: является фактором, но минимизирован использованием эффективного формата `msgspec` вместо стандартных `json` или `pickle`.
- Дополнительный узел в инфраструктуре и «новая точка отказа»: отсутствие реализации логики fallback в случае недоступности хранилища может привести к полному отказу приложения;
- Ограничения модели данных: отсутствие гибкости реляционных БД (сложные запросы, JOIN, фильтрация по полям на стороне хранилища невозможны).

#### ОБОСНОВАНИЕ И РАССМОТРЕННЫЕ АЛЬТЕРНАТИВЫ

Рассмотренные варианты:
- Redis
- Valkey
- KeyDB
- Memcached

Отклоненные варианты:
- Redis
- KeyDB
- Memcached

Основные причины:
- Redis — сменил лицензию на Source Available; его можно использовать в личных проектах, на серверах компаний и даже в коммерческих продуктах, но Redis перестал быть Open Source в юридическом смысле (OSI-compliant).
- KeyDB — форк Redis, полностью многопоточный, Open Source, но более ориентирован на нужды Snapchat (куплен Snap Inc).
- Memcached — многопоточный, максимально простой, легко настраивается, но позволяет хранить только строки (не умеет работать со структурами данных) и не имеет возможности персистентности (сохранения на диск), что критично для блэклиста refresh-токенов.

Выбор Valkey обусловлен следующими факторами:
- Основные:
  - Open Source стандарт: проект является прямым продолжением оригинального Redis при поддержке крупных игроков индустрии (AWS, Google Cloud, Oracle).
  - Поддержка протокола REdis Serialization Protocol (RESP) позволяет использовать Valkey с любыми существующими библиотеками и инструментами без правок кода (де-факто стандарт замены Redis).
- Дополнительно:
  - Производительность и инновации: включение многопоточности и оптимизаций, которые в оригинальном Redis становятся частью платных редакций.

<details>

> **Self-Note:** В программировании мы привыкли разделять str (текст) и bytes (байты). Но на уровне протоколов баз данных (Redis/Valkey/Memcached) термин String исторически означает "последовательность байтов".

> **Self-Note:** *Преимущества Valkey*
> - True Open Source: на текущий момент (апрель 2026) основная официальная альтернатива ранее свободного Redis;
> - Полная обратная совместимость с Redis: команды Redis работают в Valkey;
> - В Valkey 8.0 реализована многопоточная обработка ввода-вывода (I/O) и частичная параллелизация выполнения команд (прирост производительности на многоядерных процессорах);
> - Масштабируемость и кластеризация: улучшена работа в режиме кластера (per-slot metrics, автоматический failover);
> - Эффективность памяти: заменены старые хэш-таблицы Redis на новую структуру (Open Addressing + SIMD).
>
> **Note:** 
> - Redis использует преимущественно однопоточную архитектуру для выполнения команд. Это создает риск "Head-of-line blocking", когда выполнение тяжелой операции полностью блокирует обработку других запросов.
> - В Valkey выполнение самих команд может распределяться между потоками (в определенных сценариях), что кардинально снижает риск того, что одна "тяжелая" команда заблокирует всё хранилище.

> **Self-Note:** *4 стратегии при работе с данными (Persistence Strategies)* 
> 
> RDB - включен по умолчанию | задаем параметры ``save`` (RDB работает) или отключаем ``save ""`` (RDB выключен)
> 
> AOF - выключен по умолчанию | параметр ``appendonly no`` (AOF выключен) или ``appendonly yes`` (AOF включен)
> 
> - **RDB (Valkey Database Backup)** - это "снимки" (snapshots) всех данных в памяти, которые делаются с определенной периодичностью (например, раз в 5 минут, если изменилось 100 ключей).
>   - Плюс: Очень компактные файлы и быстрая загрузка при старте.
>   - Минус: Если Valkey упадет между снимками, потеряем данные за последние несколько минут.
>
> - **AOF (Append Only File)** - Valkey записывает каждую операцию записи в лог-файл в реальном времени.
>   - Плюс: Максимальная сохранность данных (почти нулевой риск потери).
>   - Минус: Создается дополнительная нагрузка на запись, так как файл логов постоянно растет (может вырасти то гигабайт) и в определенный момент сработает рерайт, что может привести к кратковременной нагрузке на диск и CPU.
>
> - **RDB + AOF** - AOF для надежности при перезагрузке, а RDB — для быстрых бэкапов и передачи данных на реплики.
>
> - **No Persistence** - полностью отключена запись на диск. Valkey превращается в чистый RAM-кэш. Все данные исчезают при перезапуске.

> **Self-Note:** *Настройка конфигурации для AOF параметр ``appendfsync``*
> - **appendfsync always (синхронный режим):** Вызывает fsync после каждой команды записи, максимальная надежность, но катастрофическое падение производительности.
> - **appendfsync everysec (по умолчанию, асинхронный режим):** Вызывает fsync раз в секунду, достаточно высокая скорость, запись на диск идет пачками.
> - **appendfsync no:** Системная функция fsync не вызывается Redis/Valkey, управляется ОС, максимальная скорость.
>  
> **Note:** В отличие от PostgreSQL, который по умолчанию (параметр synchronous_commit=on) гарантирует строгую долговечность (Strict Durability) через синхронную запись в WAL, Valkey (даже при включенном AOF) обычно настраивается на асинхронный сброс (appendfsync everysec).

</details>

## Выбор Reverse Proxy

#### КОНТЕКСТ
С какими проблемами мы можем столкнуться, если выставим наружу только «голый» сервер приложения:
- SSL/TLS Терминация:
  - Суть: Шифрование и расшифровка трафика - ресурсоемкая задача.
  - Последствия: Бэкенд-приложение тратит ресурсы не на обработку бизнес-логики, а на тяжелую криптографию, что может создавать избыточную нагрузку на CPU и увеличивать задержки (latency) при обработке новых соединений.
- Масштабирование:
  - Суть: Без внешнего балансировщика приложение привязано к конкретному сетевому порту одного хоста.
  - Последствия: 
    - Ограничение вертикального роста: Мы можем масштабироваться только в рамках одного узла (увеличивая число воркеров), что ограничено мощностью текущего CPU.
    - Отсутствие встроенного механизма балансировки трафика между несколькими инстансами приложения: Мы не можем запустить несколько копий (инстансов) приложения за общим IP-адресом.
    - Single Point of Failure (единая точка отказа): Если процесс сервера приложения зависнет или упадет, весь сервис станет недоступен, так как нет возможности мгновенно переключить трафик на другой рабочий инстанс.
- Терминация соединений и буферизация запросов:
  - Суть: Клиентские соединения (особенно мобильные) часто нестабильны или медленны (slow client / slowloris-like behavior).
  - Последствия: Неэффективное использование ресурсов воркеров. Granian/Uvicorn вынуждены удерживать открытые соединения с медленными клиентами в ожидании передачи тела запроса. Это может приводить к нерациональному удержанию ресурсов воркеров и снижению общей пропускной способности системы.
- Безопасность:
  - Суть: Прямое воздействие вредоносного трафика (L7 DDoS, сканирование уязвимостей, боты).
  - Последствия: Отсутствие первого эшелона защиты. Реализация Rate Limiting и фильтрации трафика на уровне приложения приводит к нецелевому расходу ресурсов воркеров. Без прокси-сервера вредоносные запросы достигают кода приложения, что увеличивает вероятность деградации бэкенда под воздействием аномального или вредоносного трафика.
- Проверки состояния и доступность:
  - Суть: Необходимость мониторинга работоспособности сервиса в реальном времени.
  - Последствия: В случае зависания или падения одного из воркеров отсутствует механизм автоматического исключения неработоспособных узлов из ротации, что ведет к росту 5xx ошибок.
- Сжатие трафика:
  - Суть: Настройка сжатия тяжелых JSON-ответов на уровне приложения (FastAPI - GZipMiddleware). 
  - Последствия: Выполнение сжатия внутри приложения создает дополнительную нагрузку на CPU, что может снижать пропускную способность бэкенда при передаче больших объемов данных.
- CORS:
  - Суть: Настройка CORS на уровне приложения.
  - Последствия: Дополнительный Middleware, нет возможности терминировать "preflight" (OPTIONS) запросы до их попадания в приложение, что приводит к выполнению лишней служебной логики на уровне бэкенда.

**Примечание (AMD FX-8320):** На архитектуре `Piledriver` задачи TLS и сжатия создают высокую конкуренцию за ресурсы в рамках общих вычислительных модулей. Вынос этих операций на уровень прокси позволяет снизить нагрузку на CPU-воркеры приложения, нивелируя риск задержек (latency spikes), которые особенно критичны при низкой производительности на одно ядро.

#### РЕШЕНИЕ
Выбран Reverse Proxy: Angie

Конфигурация:
- Инфраструктура: Делегирование задач SSL-терминации, сжатия трафика и пассивных Health Checks на уровень реверс-прокси.
- Сеть: Взаимодействие с сервером приложения (Granian) через UDS.
- Безопасность: Настройка CORS и первичных лимитов запросов (Rate Limiting) на стороне реверс-прокси.

**Примечание (Rate Limiting Strategy):** Выбрана многоуровневая защита: базовый лимитинг по IP на уровне прокси для защиты инфраструктуры и гранулярный лимитинг на уровне приложения для контроля бизнес-ресурсов (будущие задачи).

**Примечание:** Активные Health Checks доступны только в Angie PRO.

#### КОМПРОМИССЫ И ОГРАНИЧЕНИЯ
- Инфраструктурная сложность: дополнительный узел в системе, усложнение процессов деплоя и мониторинга;
- Дублирование и рассинхрон конфигурации: риск рассинхрона настроек (CORS, лимиты, таймауты) между кодом приложения и конфигурацией реверс-прокси;
- Сложный синтаксис: в отличие от Caddy или Traefik, конфигурация NGINX/Angie низкоуровневая, требует понимания работы директив;
- Ограничения UDS: потеря сетевой мобильности (требует перехода на TCP при распределении по разным узлам), отсутствие прямой отладки через curl к порту, необходимость настройки прав доступа и Shared Volumes для сокет-файла;
- Ограниченная видимость (Limited visibility): приложение «не видит» запросы, заблокированные на уровне прокси (например, по Rate Limiting или CORS);
- Усложнение жизненного цикла SSL: необходимость интеграции и поддержки сторонних утилит (mkcert, Certbot) для управления жизненным циклом сертификатов, тогда как в Caddy, Traefik этот процесс полностью автоматизирован;
- Централизация точки входа (Single Point of Failure): ошибки конфигурации прокси или некорректный reload могут повлиять на весь входящий трафик (недоступность приложения).

#### ОБОСНОВАНИЕ И РАССМОТРЕННЫЕ АЛЬТЕРНАТИВЫ

Рассматриваемые варианты:
- Caddy
- Traefik
- NGINX
- Angie

Отклоненные варианты:
- Caddy
- Traefik
- NGINX

Основные причины:
- Traefik: Автоматический SSL, высокая производительность, Auto-discovery (автоматическое обнаружение сервисов), но исторически заточен под сетевое взаимодействие (TCP/HTTP), из-за чего настройка UDS специфична, сложна и лишена гибкости.
- Caddy: Полная автоматизация SSL, максимально простой и лаконичный конфиг, отличная производительность для большинства задач, но многие низкоуровневые параметры (буферизация, тайминги, тонкий тюнинг TCP/UDS) скрыты или жестко заданы «под капотом».

**NGINX (OSS) против Angie (OSS):**
- Интерактивная панель управления (Dashboard):
  - В ванильном NGINX отсутствует визуализация и детальная статистика.
  - В Angie — веб-интерфейс со статистикой по трафику, апстримам и ошибкам.
- Экспорт метрик в JSON и Prometheus:
  - Angie может отдавать детальную статистику в формате JSON через API и напрямую в формате Prometheus, не нужно ставить отдельный nginx-exporter.
- Динамическая реконфигурация апстримов (Upstream):
  - В NGINX, чтобы добавить новый сервер в балансировщик, нужно править конфиг и делать reload.
  - В Angie — можно менять состав апстримов (добавлять/удалять сервера) и параметры (временно изменить «вес» сервера) через API без перезагрузки основного процесса.
- Поддержка HTTP/3:
  - В NGINX долгое время была в экспериментальном режиме или требовала сложной сборки с определенными библиотеками.
  - В Angie нативно «из коробки» с использованием стандартных системных библиотек (OpenSSL 3+).
- Улучшенные пассивные проверки состояния (Health Checks):
  - В отличие от NGINX, Angie позволяет не только выводить проблемные бэкенды из ротации, но и визуализировать их состояние на интерактивном дашборде.

Выбор Angie обусловлен следующими факторами:
- Основные:
   - Необходимостью работы через UDS: Traefik и Caddy поддерживают UDS ограниченно или менее гибко, в то время как Angie предоставляет полный контроль над низкоуровневой конфигурацией.
   - Требованием к тонкому контролю над сетевым поведением (buffering, timeouts, keepalive, backpressure): решения с "high-level abstraction" скрывают или ограничивают эти настройки, что затрудняет оптимизацию передачи данных между прокси и бэкендом.
   - Встроенной наблюдаемостью (метрики в формате Prometheus, JSON API) без необходимости в дополнительных компонентах.
- Дополнительно:
   - Нативной поддержкой HTTP/3, не требующей кастомной сборки или нестабильных зависимостей.

> **Self-Note:** *Роль прокси сервера и сервера приложения*
>
> **Прокси** — принимает байты из интернета, делает SSL-дешифровку и отдает их дальше.
> 
> **Сервер приложений (Granian/Uvicorn)** — принимает эти байты от прокси, превращает их в Python-словарь scope и "скармливает" приложению (FastAPI). Приложение — смотрит в scope, понимает, какой роутер вызвать, и выполняет бизнес-логику.
>
> Суть ASGI/WSGI, если упростить: мы выбираем конкретный "адаптер" (сервер приложения), который поддерживает нужный нам протокол (ASGI/WSGI), чтобы "подружить" сеть и наш синхронный/асинхронный код.
> - WSGI: Один запрос — один поток. Если запрос долгий (например, ждем ответа от БД), поток занят и не может брать другие задачи.
> - ASGI: Позволяет серверу обрабатывать тысячи соединений в одном потоке. Пока один запрос ждет данные с диска или из базы (IO-bound), Event Loop переключается на обработку другого запроса.

> **Self-Note:** *SSL/TLS Терминация (шифрование) vs Терминация соединений (TCP-сессии)*
> 
> SSL/TLS Терминация:
> - Клиент общается с прокси по зашифрованному каналу (HTTPS). Прокси расшифровывает пакеты и передает их в Granian/FastAPI уже в "чистом" виде (HTTP) через Unix-сокет/TCP.
> 
> Терминация соединений (TCP-сессии):
> - Прокси выступает как "разделитель" сессий. Он поддерживает тысячи открытых соединений с браузерами пользователей (которые могут висеть долго и ничего не слать), но при этом держит всего несколько очень быстрых и эффективных соединений с бэкендом. **Зачем?:** Создание нового TCP-соединения — это накладные расходы (рукопожатие). Прокси берет на себя "удержание" ленивых клиентов, а бэкенд нагружает только тогда, когда запрос реально готов.

> **Self-Note:** *Keep-Alive (Persistent Connection)*
> 
> Keep-Alive — это механизм, который позволяет использовать одно и то же TCP-соединение для отправки нескольких HTTP-запросов и получения ответов, не разрывая связь каждый раз.
> Как работает: После первого запроса соединение не разрывается, а остается "открытым" в режиме ожидания. Следующий запрос пойдет по тому же "каналу" мгновенно.

> **Self-Note:** *Взаимодействие Proxy <-> App: TCP vs UDS*
>
> При локальном размещении (на одном сервере) выбор протокола общения между прокси и бэкендом (Granian/Uvicorn) критически влияет на производительность.
> 
> Важное уточнение: Ядру Linux всё равно, реальная это сетевая карта (NIC) или виртуальный "петлевой" интерфейс (loopback).
> 
> - **TCP (127.0.0.1:8000):** Данные проходят через полный сетевой стек ядра. Даже на локальном хосте (loopback) это включает:
>   - Инкапсуляцию данных в TCP-пакеты.
>   - Расчет контрольных сумм для каждого сегмента.
>   - Прохождение через петлевой интерфейс.
>   - Декапсуляцию на стороне приемника.
> - **UDS (/tmp/app.sock):** Данные передаются как прямой поток байтов между процессами через буфер в ядре.
>    - Прямое копирование: Ядро просто перебрасывает данные из памяти одного процесса в память другого.
>    - Скорость: Значительно быстрее TCP за счет отсутствия проверок и упаковки в пакеты.
> 
> **Установление соединений (Handshakes & Ports)**
> - **TCP:**
>   - Процедура: Каждое новое соединение требует "рукопожатия" (**SYN -> SYN-ACK -> ACK**), если не используется Keep-Alive.
>   - Риск (Port Exhaustion): При высокой нагрузке (тысячи запросов в сек) система может исчерпать лимит эфемерных портов (Ephemeral Port Exhaustion), что приведет к отказу в новых соединениях.
> - **UDS:**
>   - Механика: Соединение устанавливается простым открытием файла сокета. Нет портов — нет лимитов на них.
>   - Безопасность: Доступ к сокету можно ограничить правами файловой системы (например, только для пользователя nginx и alexander).

</details>
